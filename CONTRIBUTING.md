## Development Setup

```bash
# 1. Fork the repo on GitHub, then clone your fork
git clone https://github.com/<your-username>/power-seo.git
cd power-seo

# 2. Add the upstream remote so you can pull updates
git remote add upstream https://github.com/CyberCraftBD/power-seo.git

# 3. Install all workspace dependencies
pnpm install

# 4. Build all 17 packages (required before running tests)
pnpm build

# 5. Run the full test suite
pnpm test

# 6. Type-check everything
pnpm typecheck

# 7. Run the linter
pnpm lint
```

### Keeping your fork up to date

```bash
git fetch upstream
git rebase upstream/main
```

### How the workspace is wired

- **Turborepo** (`turbo`) orchestrates tasks in dependency order — it knows that `@power-seo/react` depends on `@power-seo/core`, so it builds `core` first.
- **pnpm workspaces** link packages together using `workspace:*` protocol — no need to publish before cross-package imports work.
- **Changesets** (`@changesets/cli`) manages versioning and changelogs across all packages in a coordinated way.

## Package Anatomy

Every package follows an identical layout:

```
packages/<name>/
├── package.json          # Package metadata, exports map, scripts, dependencies
├── tsconfig.json         # Extends ../../tsconfig.json, sets outDir and rootDir
├── tsup.config.ts        # Bundle config — ESM + CJS + .d.ts declarations
├── vitest.config.ts      # Test runner config (node or jsdom environment)
└── src/
    ├── index.ts          # Single public API entrypoint — only export from here
    ├── types.ts          # All exported TypeScript types and interfaces
    ├── *.ts              # Implementation modules (not exported directly)
    └── __tests__/
        └── *.test.ts     # Unit tests co-located with source
```

### `package.json` export map

All packages use a standard three-condition exports map:

```json
{
  "exports": {
    ".": {
      "types": "./dist/index.d.ts",
      "import": "./dist/index.js",
      "require": "./dist/index.cjs"
    }
  },
  "main": "./dist/index.cjs",
  "module": "./dist/index.js",
  "types": "./dist/index.d.ts"
}
```

Packages with a `./react` subpath (React components) add a second entry:

```json
{
  "exports": {
    ".": { ... },
    "./react": {
      "types": "./dist/react/index.d.ts",
      "import": "./dist/react/index.js",
      "require": "./dist/react/index.cjs"
    }
  }
}
```

### `tsup.config.ts`

Standard config for non-React packages:

```ts
import { defineConfig } from 'tsup';

export default defineConfig({
  entry: ['src/index.ts'],
  format: ['esm', 'cjs'],
  dts: true,
  clean: true,
  splitting: false,
  sourcemap: true,
  treeshake: true,
  minify: false,
});
```

For React packages, add `external: ['react', 'react-dom']` to prevent bundling React:

```ts
export default defineConfig({
  entry: ['src/index.ts'],
  format: ['esm', 'cjs'],
  dts: true,
  clean: true,
  splitting: false,
  sourcemap: true,
  treeshake: true,
  external: ['react', 'react-dom'],
});
```

### `vitest.config.ts`

Use `environment: 'node'` for pure logic packages, `environment: 'jsdom'` for packages that deal with DOM APIs or render React components:

```ts
// Node (default — most packages)
import { defineConfig } from 'vitest/config';

export default defineConfig({
  test: {
    globals: true,
    environment: 'node',
  },
});

// jsdom (packages/react, packages/preview, etc.)
export default defineConfig({
  test: {
    globals: true,
    environment: 'jsdom',
  },
});
```

## Key Coding Patterns

### TypeScript strictness

The root `tsconfig.json` enables strict TypeScript — all packages inherit this:

```json
{
  "compilerOptions": {
    "strict": true,
    "noUnusedLocals": true,
    "noUnusedParameters": true,
    "noFallthroughCasesInSwitch": true,
    "noUncheckedIndexedAccess": true,
    "exactOptionalPropertyTypes": false
  }
}
```

> **`noUncheckedIndexedAccess: true`** — array indexing returns `T | undefined`, not `T`. Always guard array accesses: `const item = arr[0]; if (item) { ... }`

### No bare global references

The ESLint `no-undef` rule is active. Use `globalThis.*` for globals that may not exist in all environments:

```ts
// Wrong
const url = new URL(href);

// Correct
const url = new globalThis.URL(href);
```

### React without JSX in source

React components use `React.createElement()` instead of JSX — this avoids requiring a JSX transform in every consuming project:

```ts
// Wrong — requires JSX transform config in the consumer
export function SEOHead({ title }: Props) {
  return <head><title>{title}</title></head>;
}

// Correct — works everywhere
import React from 'react';

export function SEOHead({ title }: Props) {
  return React.createElement('head', null,
    React.createElement('title', null, title)
  );
}
```

### Tree-shaking

Every package sets `"sideEffects": false` in `package.json`. Keep it that way — never add top-level side effects (no `console.log`, no global mutations, no self-registering code at module level).

### Dual ESM + CJS

tsup produces both `.js` (ESM) and `.cjs` (CommonJS) output. Never write code that works in one format but not the other. Avoid:

- Dynamic `require()` in ESM-only paths
- Top-level `await` (not supported in CJS consumers)
- `__dirname` / `__filename` (undefined in ESM) — use `import.meta.url` and wrap with a CJS check

### Workspace dependencies

Always reference internal packages with `workspace:*`:

```json
{
  "dependencies": {
    "@power-seo/core": "workspace:*"
  }
}
```

Never hardcode a version number for a workspace package — it will drift.

### Export everything from `src/index.ts`

The `index.ts` file is the only public API surface. All types, functions, and constants must be explicitly exported from it. Anything not exported from `index.ts` is considered private.

pnpm build          # Build all 17 packages via Turborepo

## Writing Tests

Tests live in `src/__tests__/` and use [Vitest](https://vitest.dev/). The test environment is configured per-package (`node` or `jsdom`).

### Anatomy of a good test

```ts
import { describe, it, expect } from 'vitest';
import { analyzeContent } from '../index.js';

describe('analyzeContent', () => {
  it('returns a "good" status when all checks pass', () => {
    const result = analyzeContent({
      title: 'Best Coffee Shops in NYC — 2026 Guide',
      metaDescription:
        'Discover the top-rated coffee shops in New York City, from specialty roasters to hidden gems.',
      content: '<h1>Best Coffee Shops in NYC</h1><p>Long enough body text here...</p>',
      focusKeyphrase: 'coffee shops nyc',
      url: 'https://example.com/coffee-shops-nyc',
    });

    expect(result.status).toBe('good');
    expect(result.score).toBeGreaterThanOrEqual(18);
    expect(result.recommendations).toHaveLength(0);
  });

  it('returns recommendations when title is missing the keyphrase', () => {
    const result = analyzeContent({
      title: 'Great Places for Caffeine',
      metaDescription: 'A guide to New York coffee.',
      content: '<p>Short.</p>',
      focusKeyphrase: 'coffee shops nyc',
      url: 'https://example.com/test',
    });

    expect(result.recommendations.some((r) => r.toLowerCase().includes('title'))).toBe(true);
  });
});
```

### Test coverage expectations

- **Every exported function** must have at least one test.
- **Edge cases** — empty strings, empty arrays, `undefined` optional params — must be tested explicitly.
- **Error paths** — if a function throws, test that it throws the correct error type and message.
- **Pure functions** — test input/output only, no mocking needed.
- **No snapshot tests** — use explicit `expect()` assertions.

### Running a single test file

```bash
pnpm --filter @power-seo/core test -- src/__tests__/meta-validators.test.ts
```

## Code Style

### Prettier

All formatting is managed by Prettier. Do not manually adjust indentation, quotes, or trailing commas — let Prettier handle it.

```bash
pnpm format          # Format everything
pnpm format:check    # Check without writing (used in CI)
```

### ESLint

ESLint v9 flat config is used. Run before committing:

```bash
pnpm lint            # Check
pnpm lint:fix        # Auto-fix
```

Key rules in effect:

| Rule                                 | Effect                                      |
| ------------------------------------ | ------------------------------------------- |
| `no-undef`                           | Must use `globalThis.*` for globals         |
| `noUnusedLocals`                     | TypeScript removes unused local variables   |
| `noUnusedParameters`                 | TypeScript removes unused function params   |
| `@typescript-eslint/no-explicit-any` | Avoid `any` — use `unknown` or proper types |
| `react-hooks/rules-of-hooks`         | React hook rules enforced                   |

### General style guidelines

- Prefer `const` over `let`; never use `var`
- Use named exports, not default exports (makes refactoring and tree-shaking easier)
- Keep functions small and single-purpose
- Avoid `as` type assertions — fix the actual type instead
- Use `unknown` instead of `any` for untyped external data
- Write self-documenting variable names — avoid `x`, `temp`, `data`

Create branches from `main` using these prefixes:

| Prefix      | Use case                               | Example                             |
| ----------- | -------------------------------------- | ----------------------------------- |
| `feat/`     | New features                           | `feat/add-recipe-schema`            |

Keep branch names lowercase, hyphen-separated, and concise. Include the package name when the change is scoped to one package.

## Commit Conventions

We follow [Conventional Commits](https://www.conventionalcommits.org/):

```
<type>(<scope>): <description>

[optional body — explain WHY, not WHAT]

[optional footer — BREAKING CHANGE: description]
```

### Type reference

| Type       | When to use                                          |
| ---------- | ---------------------------------------------------- |
| `feat`     | A new exported function, component, or behaviour     |
| `fix`      | A bug fix                                            |
| `docs`     | README, CONTRIBUTING, or inline comment changes only |
| `refactor` | Internal restructure with no API change              |
| `test`     | Adding or updating tests                             |
| `perf`     | A change that improves performance                   |
| `chore`    | Dependency updates, config changes, build tooling    |
| `ci`       | Changes to `.github/workflows/`                      |
| `style`    | Formatting only (auto-committed by CI)               |

### Scope

The scope is the package name **without the `@power-seo/` prefix**:

```
feat(schema): add Recipe schema builder with @type and @context
fix(sitemap): handle empty URL arrays without throwing
docs(core): document validateRobots return values
test(audit): add edge case for missing h1 in body
refactor(content-analysis): extract keyword density into separate module
perf(links): replace O(n²) duplicate check with Set lookup
chore: update vitest to 3.0.0
ci: cache pnpm store between workflow runs
```

Use the root scope (`chore:`, `ci:`) when the change is not specific to one package.

### Breaking changes

Add `BREAKING CHANGE:` in the commit footer:

```
feat(redirects)!: change matchRedirect return type from boolean to RedirectMatch | null

BREAKING CHANGE: matchRedirect now returns null instead of false when no redirect matches.
Update callers: if (!matchRedirect(rules, url)) → if (matchRedirect(rules, url) === null)
```

## Creating a Changeset

Every PR that changes package behaviour (feat, fix, perf, refactor, BREAKING CHANGE) **must include a changeset**. A changeset is a small markdown file that describes what changed and at what semver bump level.

```bash
# In the repo root
pnpm changeset
```

The interactive prompt will ask:

1. **Which packages changed?** — Select with spacebar, confirm with Enter
2. **Major / Minor / Patch?** — Choose the correct semver level:
   - `patch` — bug fix, internal refactor, no API change
   - `minor` — new exported function or option, backwards-compatible
   - `major` — breaking API change (removed export, changed signature)
3. **Describe the change** — A one-sentence summary for the changelog

A `.changeset/<random-name>.md` file is generated. Commit it with your PR:

```bash
git add .changeset/
git commit -m "chore: add changeset for feat/add-recipe-schema"
```

### What not to add a changeset for

- `docs/` — README or CONTRIBUTING changes only
- `test/` — test changes that don't affect published behaviour
- `ci/` — workflow changes
- `chore/` — dependency updates that don't affect the public API

## Pull Request Process

### Before opening a PR

Run the full check suite locally to avoid CI failures:

```bash
pnpm build       # Must succeed for all packages
pnpm typecheck   # Must pass with zero errors
pnpm lint        # Must pass (or run lint:fix first)
pnpm test        # All tests must pass
pnpm format:check  # Must pass (or run format first)
```

### Steps

1. **Create a branch** from `main` using the naming conventions above
2. **Make focused changes** — one concern per PR; avoid mixing unrelated fixes
3. **Add a changeset** if the PR changes package behaviour (see above)
4. **Write or update tests** for any new or changed functionality
5. **Push and open the PR** — use the PR title in Conventional Commits format:
   ```
   feat(schema): add Recipe and HowTo schema builders
   ```
6. **Fill in the PR description** — describe what changed and why, link any related issues
7. **Wait for CI** — the `Production` workflow runs build, type-check, and lint
8. **One approval required** — a maintainer will review and either approve or request changes
9. **Do not force-push** after review begins — add new commits instead

### PR size guidelines

- **Small PRs merge faster.** Aim for under 400 lines of change.
- If a feature is large, split it: infrastructure first, then features, then tests.
- Refactoring PRs should contain zero behaviour changes — make them easy to review.

## Adding a New Package

Before implementing, open an issue and describe the package. A new package must:

- Not duplicate functionality already covered by an existing package
- Be independently useful (not force-installed with another package)
- Fit the TypeScript-first, zero-runtime-dependency philosophy

### Step-by-step

**1. Create the package directory**

```bash
mkdir packages/<name>
```

**2. Add `package.json`**

Use an existing package as a template. The key fields:

```json
{
  "name": "@power-seo/<name>",
  "version": "1.0.0",
  "description": "One sentence description",
  "license": "MIT",
  "type": "module",
  "exports": {
    ".": {
      "types": "./dist/index.d.ts",
      "import": "./dist/index.js",
      "require": "./dist/index.cjs"
    }
  },
  "main": "./dist/index.cjs",
  "module": "./dist/index.js",
  "types": "./dist/index.d.ts",
  "files": ["dist"],
  "scripts": {
    "build": "tsup",
    "dev": "tsup --watch",
    "test": "vitest run",
    "test:watch": "vitest",
    "typecheck": "tsc --noEmit",
    "lint": "eslint src/",
    "lint:fix": "eslint src/ --fix",
    "clean": "rimraf dist"
  },
  "devDependencies": {
    "rimraf": "^6.0.0",
    "tsup": "^8.3.0",
    "typescript": "^5.7.0",
    "vitest": "^2.1.0"
  },
  "sideEffects": false,
  "author": "CyberCraft Bangladesh <info@ccbd.dev>",
  "repository": {
    "type": "git",
    "url": "git+https://github.com/CyberCraftBD/power-seo.git",
    "directory": "packages/<name>"
  },
  "publishConfig": {
    "access": "public"
  }
}
```

**3. Add `tsconfig.json`**

```json
{
  "extends": "../../tsconfig.json",
  "compilerOptions": {
    "outDir": "./dist",
    "rootDir": "./src"
  },
  "include": ["src"],
  "exclude": ["node_modules", "dist", "**/__tests__/**"]
}
```

**4. Add `tsup.config.ts`**

Copy from `packages/core/tsup.config.ts` (or `packages/react/tsup.config.ts` for React packages).

**5. Add `vitest.config.ts`**

Use `environment: 'node'` unless the package deals with the DOM.

**6. Implement the public API**

```
src/
├── index.ts      # Re-export everything public
├── types.ts      # All exported types
└── <module>.ts   # Implementation
```

**7. Write tests**

At minimum: one happy-path test and one edge-case test per exported function.

**8. Write a README.md**

Follow the same documentation standard as the other packages — see `packages/audit/README.md` or `packages/links/README.md` as reference. The README must include: quick start, installation, API reference, and examples.

**9. Link the workspace**

```bash
pnpm install   # Run from the repo root to link the new workspace package
```

**10. Build and verify**

```bash
pnpm --filter @power-seo/<name> build
pnpm --filter @power-seo/<name> test
pnpm --filter @power-seo/<name> typecheck
```

**11. Update the root README**

Add the new package to the packages table and the ecosystem table at the bottom.

**12. Add a changeset**

```bash
pnpm changeset
# Select the new package → minor (new feature) → describe it
```

## CI/CD Pipeline

The `production.yml` workflow runs on every push to `main`. It is **not** a traditional test-and-gate CI — it is a full release pipeline:

| Step                | What it does                                                       |
| ------------------- | ------------------------------------------------------------------ | --- | ------ |
| **Checkout**        | Full history clone (`fetch-depth: 0`) for correct versioning       |
| **Install**         | `pnpm install --frozen-lockfile` — lockfile must match             |
| **Build**           | `turbo run build --filter='./packages/*'`                          |
| **Type-check**      | `turbo run typecheck --filter='./packages/*'`                      |
| **Auto-fix lint**   | `turbo run lint:fix` (failures are non-blocking — `                |     | true`) |
| **Auto-fix format** | `pnpm format` (non-blocking)                                       |
| **Commit fixes**    | If lint/format changed files, commits them with `[skip ci]`        |
| **Bump versions**   | Bumps all packages to the next patch version                       |
| **Publish to npm**  | Publishes each package if the version doesn't already exist on npm |
| **Commit + tag**    | Commits version bumps, creates a git tag `vX.Y.Z`                  |
| **GitHub Release**  | Creates a GitHub Release with install instructions                 |

> The `[skip ci]` tag in auto-fix commit messages prevents infinite loops.

### What this means for contributors

- **Do not push directly to `main`** without a working build and type-check.
- **Lint errors do not block the pipeline** — the CI auto-fixes them. But fix them locally first: `pnpm lint:fix`.
- **Version bumps happen automatically** — you do not need to bump `package.json` versions in your PR. The CI does it.
- **Every merge to main triggers a publish.** Make sure your changes are ready before merging.

## Versioning Policy

`@power-seo` follows [Semantic Versioning](https://semver.org/):

| Change type                     | Version bump    | Examples                                      |
| ------------------------------- | --------------- | --------------------------------------------- |
| Bug fix, internal refactor      | `patch` (1.0.x) | Fix a broken regex, improve performance       |
| New exported function or option | `minor` (1.x.0) | Add `buildFaqPrompt()`, add optional param    |
| Breaking API change             | `major` (x.0.0) | Remove an export, change a function signature |

### Stability guarantees

- All packages are on `v1.x.x` — **the public API is stable**.
- No existing exported function or type will be removed without a major version bump.
- No package will ever be moved behind a paywall. The 17 packages are MIT-licensed forever.

### `workspace:*` vs published versions

During development, packages reference each other with `workspace:*`. When published, pnpm replaces this with the exact version number that was published.

## Security Policy

### Reporting a vulnerability

Do **not** open a public GitHub issue for security vulnerabilities. Contact us privately:

**Email:** [info@ccbd.dev](mailto:info@ccbd.dev)
**Subject:** `[SECURITY] @power-seo — <short description>`

Include:

- The package name and version affected
- A description of the vulnerability
- Steps to reproduce
- Potential impact

We aim to acknowledge reports within 48 hours and publish a fix within 14 days for confirmed vulnerabilities.

### Dependency vulnerabilities

Transitive dependency vulnerabilities in `devDependencies` are addressed via `pnpm.overrides` in the root `package.json` where a safe patch-level bump is available. Vulnerabilities in production dependencies are treated as bugs and fixed immediately.
